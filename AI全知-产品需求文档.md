# AI全知 — 产品需求全景文档

> **产品定位**：AI 时代的 GitHub + Stack Overflow + ArXiv + ProductHunt 融合体，以 AI 为第一公民、人类与 AI 共生协作的知识与技能聚合平台。
>
> **一句话**：一个和 AI 行业同步进化的、以动手实践为核心的、大模型驱动的 AI 技能学习平台。
>
> **文档版本**：v2.0 | **创建日期**：2026-05-23 | **更新日期**：2026-05-23

---

## 一、核心价值主张

### 三个核心信念

1. **知识不该有滞后** — 新东西出来，24 小时内形成结构化、可实践的知识单元，不等博主、不等翻译、不等教程
2. **讨论是知识的生产方式** — 人类和 AI 共同讨论，讨论过程本身就是知识，不只是评论区的附属品
3. **学习就是动手** — 每一个知识点都能一键进入实践环境，不是"懂了"，是"跑通了"

### 平台核心原则

| 原则 | 说明 |
|------|------|
| **速度与质量双保障** | 平台大部分决策由 AI 驱动，但引入三 AI 质量讨论机制确保准确性 |
| **来源驱动而非 AI 驱动** | 新闻聚合筛选优先依靠代码公式与来源权威性，AI 辅助但不主导 |
| **AI 中立被召唤** | AI 以中立角色参与讨论，仅在被召唤时发言，不主动发起 |
| **公开与隐私分层** | 知识库公开，讨论公开，用户数据严格隐私保护 |
| **知识优先于社区** | 冷启动阶段知识库建设优先，社区内容可从零开始，知识库必须有数据 |

### 与传统平台的本质区别

| 维度 | 传统方案 | AI全知 |
|------|---------|--------|
| 内容生产 | 人写教程，发布即开始过时 | AI+人协作，持续更新，发布即开始生长 |
| 知识形态 | 文章/视频，静态 | 结构化知识单元 + 关联图谱，可计算 |
| 热点处理 | 新闻列表，看完就忘 | 热点是知识图谱的更新触发器，不是终点 |
| 评测方式 | benchmark表格，脱离场景 | 场景化、持续化、可质疑、可参与 |
| 社区角色 | 评论区，附属品 | 讨论是核心功能，讨论本身就是知识生产 |
| AI 角色 | 工具（帮你写、帮你搜） | 中立的被召唤者，非主动参与者 |
| 学习方式 | 看教程→照抄→完成 | 设定目标→动态路径→动手挑战→AI反馈→成长可量化 |
| 时效性 | 滞后（等博主写教程） | 24小时内结构化知识单元生成 |
| 个性化 | 推荐算法（你喜欢啥推啥） | 知识画像（你不知道啥推啥，你该知道啥推啥） |
| 质量控制 | 编辑审核 | 三 AI 对抗讨论 + 代码公式 + 来源权重体系 |

## 二、用户角色与需求映射

### 用户分层

| 角色 | 核心场景 | 核心诉求 |
|------|---------|---------|
| **探索者**（入门者） | "我想学 AI，不知道从哪开始" | 可信赖的学习路线、难度自适应、避免踩坑 |
| **实践者**（工程师） | "这个新东西我能不能用到项目里" | 快速评测、部署可行性、成本计算、踩坑经验 |
| **决策者**（技术负责人） | "团队该投哪个方向" | 技术趋势判断、竞品对比、迁移风险评估 |
| **布道者**（内容创作者） | "我有经验想分享" | 知识贡献获得认可、AI 辅助写作、内容影响力 |
| **探路者**（研究者） | "前沿在发生什么" | 模式识别、跨界启发、非共识发现 |

### AI 也是用户

平台上的 AI Agent 不是后台工具，而是**显式的参与者**，但遵循严格的行为边界：

- AI 可以发布内容（标注"AI 生成"）
- AI 以**中立角色**参与讨论，不持有主观立场
- AI **仅在被召唤时发言**，不主动发起讨论
- AI 可以根据用户反馈调整未来行为
- 用户可选择在某次讨论中召唤或静默 AI

## 三、核心功能模块

### 模块 1：热点雷达 — 代码化质量控制的信号捕获系统

#### 核心原则：来源驱动，而非 AI 驱动

新闻聚合筛选**不能只依靠 AI**。必须建立一套"代码公式 + 来源权重"的双轨质量控制体系。

#### 1.1 数据源分级与权重体系

每个数据源有确定的权重，由代码计算而非 AI 判断：

| 级别 | 权重区间 | 典型来源 | 判定依据 |
|------|---------|---------|---------|
| **S 级**（一手权威） | 90-100 | ArXiv 论文原文、官方技术报告、GitHub 官方仓库 Release、公司官方 Blog | 信息源头，无需交叉验证 |
| **A 级**（二手权威） | 70-89 | HuggingFace Daily Papers、Papers With Code、学术会议官网、权威科技媒体（TechCrunch/VentureBeat） | 可靠，但有编辑加工 |
| **B 级**（优质社区） | 50-69 | Hacker News 首页、Reddit r/MachineLearning 高票帖、GitHub trending、知名技术博客 | 有信息量，需交叉验证 |
| **C 级**（广泛社区） | 30-49 | Twitter/X 知名账号、知乎高赞、即刻、公众号 | 观点/讨论价值，事实需验证 |
| **D 级**（参考级） | 10-29 | 普通自媒体、个人博客、未知来源 | 仅作参考，不纳入正式知识库 |

#### 1.2 信噪比计算公式

每条热点资讯进入系统时，由代码计算信号强度：

```
信号强度 = (来源权重 × 0.4) + (多源交叉验证数 × 0.25) + (实体关联密度 × 0.2) + (时效性衰减系数 × 0.15)

其中：
- 多源交叉验证数 = 在时间窗口内不同来源独立报道同一事件的来源数量（去重后），上限为5
- 实体关联密度 = 该资讯关联到的知识库已有实体数量 / 资讯长度归一化值，上限为1
- 时效性衰减系数 = e^(-λt)，t为发布时间距今小时数，λ=0.01（约7天后衰减至0.2）
```

**信号强度阈值**：
- ≥ 0.70：高价值热点，自动进入技术报告生产队列
- 0.45 - 0.69：潜在热点，进入待观察池，继续积累信号
- < 0.45：降级为普通资讯，仅聚合不深度处理

#### 1.3 数据源矩阵

| 类型 | 来源 | 捕获策略 | 质量保障 |
|------|------|---------|---------|
| 学术前沿 | ArXiv, Papers With Code, OpenReview, 顶会论文集 | 新论文 → 结构化提取 → 关联已有知识库 → 判定新鲜度 | 以 ArXiv ID 和 DOI 做唯一标识，按引用量/复现状态加权 |
| 开源动态 | GitHub trending, GitLab, Hugging Face, ModelScope | 新项目/新release → 自动分析README/代码结构 → 生成能力卡片 | 按 Star/Fork/活跃度/Pulse 计算开源健康度，非 AI 判定 |
| 行业新闻 | TechCrunch, VentureBeat, The Verge, The Information | 事件抽取 → 结构化 → 关联受影响的技术栈 | 仅选用 S/A 级来源的事实报道，排除观点/评论类内容 |
| 产品发布 | ProductHunt, Hacker News Show, 各AI产品官网Blog | 新产品/新功能 → 自动生成产品卡片 + 同类对比 | 以官方 Changelog 为唯一事实锚点 |
| 社区讨论 | Reddit (r/ML, r/LocalLLaMA), Hacker News, 知乎, 即刻, Discord精华 | 热度检测 → 观点聚类 → 提取有信息量的讨论节点 | 仅提取事实性信息和可验证观点，过滤纯情绪内容 |
| 官方动态 | 各大AI公司Blog, Changelog, API文档变更, 模型卡片 | 版本变更 → 影响分析 → 迁移指南自动生成 | 官方源为最高优先级，任何冲突以官方源为准 |
| 自媒体内容 | YouTube（技术向频道）, 公众号, Substack, 独立博客 | 内容聚合 → 关键观点提取 → 交叉验证 → 去重 | D 级来源，仅作参考和观点补充，不进入知识库事实层 |

#### 1.4 信号处理流水线

```
原始资讯抓取
    ↓
来源识别与权重赋值（代码规则，非AI）
    ↓
信噪比公式计算（代码公式）
    ↓
┌─ 高信号(≥0.70) → 进入快速通道 → 自动分类 → 技术报告工厂
├─ 中信噪(0.45-0.69) → 观察池 → 多AI交叉讨论 → 等待更多信号
└─ 低信号(<0.45) → 降级聚合 → 信息流展示 → 不入库
    ↓
多源去重与事件合并（实体识别+时间窗口聚类）
    ↓
关键词自动提取与分类（AI辅助，代码校验）
    ↓
入库 → 触发关联图谱更新
```

#### 1.5 关键词自动分类体系

AI 对每篇文章/技术/产品自动提取关键词并分类，分类结果由代码规则校验：

**分类维度**（必分，不可遗漏）：

| 分类维度 | 可选值 | 分类方式 |
|---------|--------|---------|
| **实体类型** | 模型 / 工具框架 / 论文 / 产品 / 范式方法 / 事件 / 人物 | AI 分类 + 代码特征校验 |
| **技术领域** | LLM / 多模态 / Agent / RAG / Embedding / 推理优化 / 数据工程 / 评测 / 对齐安全 / 部署推理 / 其他 | AI 关键词提取 + 预定义词典匹配 |
| **应用层级** | 基础研究 / 基础设施 / 开发工具 / 应用产品 / 方法论 | 代码规则为主 |
| **影响范围** | 全行业 / 特定领域 / 局部 | 来源权重 + 讨论量 |
| **可落地性等级** | 即刻可用 / 近期可用(1-3月) / 需观望(3-6月) / 纯研究 | 三 AI 讨论后判定 |
| **信息类型** | 事实报道 / 技术分析 / 观点评论 / 教程指南 / 官方公告 | 代码规则 + 来源类型 |

### 模块 2：三层 AI 质量分析体系 — 内容质量的制度性保障

这是平台最核心的质量控制机制。所有 AI 生成的内容（技术报告、评测、摘要）在发布前，必须经过**三个独立 AI 角色的对抗性讨论**。

#### 2.1 三角色定义

| 角色 | 定位 | 行为准则 |
|------|------|---------|
| **生成者 (Generator)** | 内容创建者 | 负责从原始信息生成结构化内容，追求全面和准确 |
| **质疑者 (Challenger)** | 专业批判者 | 系统性地寻找事实错误、逻辑漏洞、信息遗漏、过度推断。强制要求引用来源 |
| **仲裁者 (Arbiter)** | 平衡裁决者 | 对比双方论点，指出事实 vs 观点的区别，判定哪些争议需要标注、哪些需要重新验证 |

#### 2.2 质量讨论流程

```
[原始信息]
    ↓
生成者 → 产出初稿（技术报告/评测/摘要）
    ↓
质疑者 → 逐条审查，列出质疑清单（含来源引用）
    ↓
生成者 → 回应质疑，修改或反驳（需提供理由）
    ↓
质疑者 → 再次审查修改结果
    ↓
仲裁者 → 裁决未解决的争议点
    ├─ 无争议 → 通过，发布
    ├─ 有争议但可标注 → 标注"存在争议：[具体争议点]"，通过
    └─ 重大争议 → 标记为"待验证"，暂不发布，等待更多信息来源
    ↓
最终输出：内容 + 质量分 + 争议标注 + 置信度
```

#### 2.3 质量评分体系

每次三 AI 讨论后，仲裁者给出质量评分：

| 评分维度 | 权重 | 评分标准 |
|---------|------|---------|
| **事实准确度** | 35% | 是否有可验证的来源支撑？是否存在已知事实错误？ |
| **信息完整度** | 20% | 是否遗漏了重要信息？多源信息是否充分采集？ |
| **逻辑一致性** | 15% | 结论与论据之间是否存在逻辑断裂？ |
| **客观中立性** | 15% | 是否存在未标注的主观判断？是否过度推断？ |
| **时效性标注** | 10% | 是否标注了信息的时效性限制？是否说明了可能变化的点？ |
| **可操作性** | 5% | 对实践者是否有明确的行动指导？ |

**质量等级**：
- **S 级（≥90分）**：高度可信，可作为决策依据
- **A 级（80-89分）**：可信，有少量标注的争议点
- **B 级（70-79分）**：基本可信，存在需注意的不确定性
- **C 级（60-69分）**：参考级，信息不完整或争议较多
- **D 级（<60分）**：不可发布，需重新生成

#### 2.4 质量体系的透明度

- 每篇 AI 生成内容的顶部展示：质量等级 + 三位 AI 的讨论摘要（可展开查看完整讨论记录）
- 用户可以查看"质疑者提出了什么、生成者如何回应、仲裁者如何裁决"
- 用户可以对 AI 的讨论质量打分，反馈回训练体系

### 模块 3：技术报告工厂 — 质量受控的深度内容生产

当新模型/新框架/新工具发布时，系统自动或半自动生成结构化技术报告。所有报告必须通过三层 AI 质量审查。

#### A. 新模型技术报告

- **架构解读**：用统一框架重新解读，不是复制技术报告
- **性能实测**：在平台标准评测集上跑一遍，不是引述官方数据
- **部署指南**：Ollama/vLLM/TGI/本地，多种方案 + 硬件要求 + 成本估算
- **场景能力矩阵**：代码/推理/多语言/长文本/多模态 → 每个维度的实测评分
- **竞品对比雷达图**：与同级别模型的横向对比
- **"我该用它吗"决策树**：根据用户场景给推荐

#### B. 新框架/工具技术报告

- **核心抽象解析**：核心理念是什么，怎么理解最快
- **最小可用示例**：5分钟跑通的最小闭环
- **与现有方案的差异**：不只是功能列表，是设计哲学的差异
- **集成路径**：如何接入5种主流技术栈
- **第一天踩坑清单**：由社区贡献 + AI自动发现
- **维护健康度评估**：commit频率、issue响应、社区活跃度、依赖风险

#### C. 新范式/方法论报告

- **背景**：什么背景下提出的，为什么需要
- **核心思想**：一句话 + 三分钟 + 深入版，三级理解层次
- **适用边界**：什么场景下有用，什么场景下别用
- **与其他范式的关系**：互补/替代/无关
- **实践指南**：从零开始的阶梯式教程，每个步骤可执行

#### D. 新产品/平台评测报告

- **产品能力拆解**：不是官网介绍搬运，是实际使用后的结构化分析
- **定价与性价比分析**：做什么事要花多少钱
- **上手体验**：注册到第一个成果的时间、步骤、卡点
- **生态锁定风险**：数据迁移难度、API依赖性、竞品可替代性
- **团队适配度**：什么样的团队规模/技术栈最适合用

#### 报告生命周期

- **第一版（24小时内）**：AI生成 + 三AI质量审查 → 标注"AI初稿，质量等级X"
- **第二版（72小时内）**：社区讨论补充，补充实测数据和个人经验 → 标注"社区补充版"
- **第三版（持续迭代）**：技术演进带来的更新 → 报告有"版本号"和"更新日志"
- **过期标记**：当上游技术发生变化时，报告自动标记"部分内容可能已过时"并标注过期部分

### 模块 4：知识基因库 — 结构化的知识图谱

AI全知的底层数据模型——不只是文章，而是**可计算的知识单元**。

#### 核心实体

```
Model (模型)
  ├─ 架构类型, 参数规模, 模态能力, 上下文长度
  ├─ 训练数据, 许可证, 开放程度
  ├─ 推理速度, 部署门槛, 成本
  ├─ 发布方, 发布时间, 版本历史
  └─ 关联: [Paper] [Benchmark结果] [部署方案] [替代/互补模型]

Tool (工具/框架)
  ├─ 定位, 核心抽象, 语言生态
  ├─ 许可证, 维护方, Star/Fork趋势
  ├─ 依赖, 集成生态, 学习曲线
  └─ 关联: [教程] [对比评测] [迁移指南] [配套模型]

Skill (技能/范式)
  ├─ 难度级别, 前置技能, 预计学习时间
  ├─ 适用场景, 不适用场景
  ├─ 成熟度, 变化速度, 半衰期
  └─ 关联: [学习路径] [实践挑战] [相关工具] [经典案例]

Paper (论文)
  ├─ 核心贡献, 方法类型, 实验设置
  ├─ 引用量/趋势, 复现状态
  └─ 关联: [相关论文] [实现代码] [衍生工具] [批评/讨论]

Product (产品)
  ├─ 功能领域, 目标用户, 定价模型
  ├─ 技术栈, 集成能力, 竞品关系
  └─ 关联: [评测报告] [用户评价] [替代方案]

Scenario (应用场景)
  ├─ 行业, 任务类型, 难度
  ├─ 典型解决方案, 效果基准
  └─ 关联: [相关模型] [相关工具] [案例] [挑战]

Person/Org (人物/组织)
  ├─ 角色, 所属机构, 研究/产品方向
  ├─ 影响力范围
  └─ 关联: [发布的项目/论文] [观点/言论] [合作关系]
```

#### 关键能力

- **跨实体关联**："一个争议性论文 → 引发的社区讨论 → 催生的新工具 → 被哪个模型实现 → 相关的实践挑战"
- **时间轴视图**：任何知识点可追溯"它之前发生了什么导致了它，它之后发生了什么改变了它"
- **关系强度量化**：不只是"关联"，而是"多强相关"——是核心影响还是边缘提及

### 模块 5：自由广场 — AI 中立的被召唤式社区

#### 5.1 AI 参与社区的铁律

| 规则 | 内容 |
|------|------|
| **只能被召唤** | AI 绝不主动发言。仅在用户 @AI角色 时回应。不在未被召唤的讨论中出现 |
| **保持中立** | AI 不持有主观立场。可以提供正反两方的论据，但不对争议做价值判断 |
| **明牌身份** | 所有 AI 发言必须明确标注"AI"身份和具体角色，不可拟人化 |
| **来源必引** | AI 的每一个事实性陈述必须附带可验证的来源链接 |
| **置信度必标** | AI 的每一个判断/推断必须标注置信度（确信/可能/推测） |
| **可被质疑** | 任何用户可以对 AI 的发言提出质疑，触发 AI 重新验证 |

#### 5.2 内容形态：观点单元

每条内容（人发的或AI发的）都是结构化对象：

- **论点**：一个可被讨论、验证、挑战的声明
- **论据**：支持论点的信息（内链到知识库实体、实测数据、外部源）
- **立场**：在争议上的位置（细粒度的态度光谱，不是简单的赞同/反对）
- **置信度**：作者的确信程度（"我确信" / "我猜测" / "听说"）

#### 5.3 可被召唤的 AI 角色

| AI 角色 | 召唤方式 | 行为 | 限制 |
|---------|---------|------|------|
| **苏格拉底** | @苏格拉底 | 追问"为什么"、"条件变了会怎样"、"有反例吗" | 仅提供问题，不提供答案 |
| **连接者** | @连接者 | 关联知识库中相关但未被引用的历史内容 | 仅提供链接和摘要，不做判断 |
| **翻译官** | @翻译官 | 术语翻译为入门语言，或将模糊直觉翻译为技术语言 | 不添加新信息，仅翻译表达 |
| **事实核查员** | @核查员 | 验证引用的准确性，标记事实与观点的区别 | 仅核查可验证的事实，不评价观点 |
| **魔鬼代言人** | @代言人 | 系统性地为少数派观点辩护 | 必须声明"以下为少数派观点" |
| **趋势观察员** | @观察员 | 跨讨论模式识别，提出可能存在的趋势 | 标注为"模式观察，非确定性结论" |
| **知识管家** | @知识管家 | 将讨论中有价值的知识点提取、结构化 | 需经参与者确认后才可录入知识库 |

#### 5.4 讨论形态

**（1）技术报告下的讨论**
- 每份技术报告自动生成"讨论空间"
- 讨论按维度组织：性能实测区、部署体验区、场景适配区……
- AI 不主动引导，仅在用户召唤时出现

**（2）热点事件下的"无边界讨论"**
- 不只是评论区，而是一个**讨论空间**
- 支持分支讨论：人类用户可开启子话题
- 讨论可视化：讨论树/讨论地图，可看到结构、关键节点、共识和分歧位置
- 讨论"成果化"：讨论结项时（由参与用户发起），召唤@知识管家提取要点

**（3）"对撞实验室"**
- 两个观点对立的**人类用户**进入结构化辩论
- 观众可投票、提供新证据、加入任一方
- 辩论结束后，可召唤 AI 生成"辩论报告"——双方论据、逻辑链、未解决的问题
- AI 不参与辩论本身，仅做记录和整理

**（4）"方案擂台"**
- 面对一个任务，社区提出不同实现方案
- 每个方案包含：技术选型、代码仓库、成本估算、优劣分析
- 社区投票带场景标签（"我在这个场景下会选这个"）
- AI 擂台裁判（需召唤）：评测每个方案的可运行性、性能、成本

#### 5.5 社区治理原则

- **AI 不可隐身**：所有 AI 行为必须明确标注，不冒充人类
- **AI 不可主动**：AI 不能发起话题，不能刷屏，不能主导讨论节奏
- **人类优先**：讨论的节奏和方向由人类控制，AI 是工具而非参与者
- **自由不等于无政府**：通过声誉系统和社区规范自治，不是中心化审查
- **AI 中立不可动摇**：AI 不选边、不站队、不持有立场

### 模块 6：技能工厂 — 可执行的学习路径

#### 6.1 技能树

- 动态更新的 AI 技能树（可视化图展示）
- 每个节点是一个技能，节点大小反映当前热度/需求度，颜色反映成熟度
- 点击任何节点进入该技能的学习页面

#### 6.2 学习路径

- 用户设定目标 → 系统生成路径（考虑用户当前水平 + 最新技术栈）
- 路径不是静态的，会"变质"——某个环节已过期时自动提醒并提供替代
- 每个节点不是"看完这篇教程"而是"完成这个挑战"（通过才能继续）
- 支持"曲线救国"——如果卡住了，AI 推荐另一条可能更适合的学习路线

#### 6.3 实践挑战

- **沙箱环境**：直接在浏览器中运行代码（云端沙箱），不需要本地配置
- **验证标准**：不只是代码能跑，而是要达成特定结果
- **AI 助教**：卡住时提供递进式提示（方向 → 提醒 → 部分答案 → 完整方案）
- **闯关记录**：完成时间、求助次数、代码质量——用来优化挑战难度和用户能力模型

#### 6.4 技能债务追踪

- 学过的技能后来变化了（API breaking / 范式过时 / 更好的替代出现）
- 系统推送"技能债务提醒"
- 用户技能档案显示：哪些还是新鲜的，哪些已经"过期"，哪些正在贬值

### 模块 7：落地可行性自动评测系统

#### 7.1 评测流水线

**阶段 1：信息收集**
- 抓取所有公开信息：官方文档、技术报告、GitHub、博客、社交媒体讨论
- 重点不是"官方说了什么"，而是"社区发现了什么问题"

**阶段 2：自动化验证**
- 模型可用时在沙箱中自动跑标准评测集
- 评测集由社区贡献，贴近真实场景
- 记录：部署耗时、资源消耗、成功率、输出质量、响应延迟
- 开源代码跑一遍 demo，验证 README 是否能复现

**阶段 3：多维评分**

| 维度 | 评估内容 | 自动化方式 |
|------|---------|-----------|
| 技术成熟度 | API稳定性、文档质量、bug密度 | 自动跑集成测试、分析issue tracker、检测文档覆盖率 |
| 部署门槛 | 对硬件/基础设施的最低要求 | 在标准化沙箱中实际部署，记录资源消耗和步骤数 |
| 性能基准 | 在特定场景下的真实表现 | 跑用户定义的评测集，不是公开benchmark |
| 生态兼容 | 与主流框架/工具的互操作程度 | 自动检测与LangChain/LlamaIndex/Haystack等的集成状态 |
| 社区健康度 | 实质活跃度而非star数 | PR合并速度、issue关闭率、贡献者多样性 |
| 成本效益 | 价格/性能/质量三角 | 自动跑相同任务对比不同方案的成本和效果 |

**阶段 4：生成推荐**
- 系统根据评分和用户画像，生成个性化推荐："应该关注" / "应该试试" / "应该等等" / "可以忽略"

#### 7.2 持续评测

- 不是一次性快照，每个被评测对象有"评测历史"时间线和分数变化趋势
- 异常检测：某个模型突然变差 → 触发重新评测

#### 7.3 评测可信度

- AI自动评测标注为"AI评测"，可信度取决于评测方法
- 鼓励社区贡献评测（标注"人工评测"）
- 多个独立评测来源互相印证
- "评测的评测"：社区可以对评测本身打分/质疑

#### 8.1 场景库

持续积累和归类真实应用场景：

| 金融 | 研报总结、风险分析、合规审查、智能投顾 | 中 |

#### 8.3 产品技能聚合

- 为每个产品维护"技能地图"——这个产品现在能做什么、不能做什么
- "新技能发现"：当社区发现产品的"隐藏技能"或"非预期用法"时收录并关联到场景
- "技能移植"：产品A有的功能，产品B能不能做到？怎么做？→ 自动生成对比/替代方案

### 模块 9：个人知识库管理 — 用户的知识私有空间

这是用户自己的知识管理空间。用户可以没有，社区内容可以没有，但每个注册用户都有自己独立的、完全自主控制的知识库。

#### 9.1 个人知识库与公共知识库的关系

| 维度 | 公共知识库 | 个人知识库 |
|------|----------|----------|
| 所有权 | 平台公共 | 用户私有 |
| 可见性 | 全网可见 | 仅自己可见（默认），可选公开 |
| 内容来源 | AI自动聚合 + 社区贡献 | 用户自行创建 + 从公共库收藏 |
| 编辑权 | 社区协作 + AI辅助 | 完全用户自主 |
| 隐私保护 | 不涉及个人数据 | 严格加密，不可被AI或其他用户访问 |

#### 9.2 核心功能

**（1）知识笔记编辑器**
- 富文本 + Markdown 编辑器，支持代码高亮、数学公式、Mermaid 图表
- 结构化模板：论文笔记模板、模型评测模板、工具上手模板、技术调研模板
- AI 写作辅助（由用户召唤，非自动）：润色、扩写、翻译、总结

**（2）从公共库一键收藏**
- 浏览公共知识库时，一键收藏任何知识单元到个人空间
- 收藏时可添加个人批注、标签、关联自己的其他笔记
- 收藏项保持与公共库的链接，当公共库更新时收到"源内容已更新"通知

**（3）个人知识图谱**
- 用户的个人知识自动构建关联网络
- 可视化展示：我关注哪些领域、我的知识密度分布、我的知识空白区
- 与公共知识图谱对比：哪些热点我还没关注到？

**（4）标签与分类体系**
- 用户自定义多级分类目录
- 智能标签推荐（AI 根据内容自动建议标签，用户决定是否采用）
- 全文检索 + 标签筛选 + 时间筛选 + 关联检索

**（5）学习笔记与进度追踪**
- 完成平台上的实践挑战后，自动生成学习记录存入个人知识库
- 用户在技能工厂的学习路径和进度可视化
- 学习笔记与公共知识库的双向链接

**（6）知识导出**
- 支持导出为 Markdown、PDF、JSON
- 支持批量导出
- 用户数据永远可带走，不做锁定

**（7）个人知识库发布（可选）**
- 用户可选择将个人笔记发布到公共社区
- 发布后经过三 AI 质量讨论，标注"社区贡献"来源
- 保留原作者署名和原始链接

#### 9.3 数据隐私保障

- 个人知识库内容加密存储，密钥由用户控制
- 平台管理员无法查看用户私有内容
- AI 仅在用户明确召唤时才能访问个人知识库内容
- 用户的浏览历史、学习记录、兴趣画像均为私有不公开数据
- 隐私设置面板：用户可以逐项控制哪些数据可以被用于什么目的

### 模块 10：个人 AI 画像 — 知识与能力的量化

#### 10.1 知识图谱
- 系统为每个用户维护个人知识图谱：你会什么、到什么程度

#### 10.3 学习历史与趋势

- 哪些技能在上升，哪些在搁置
- 与行业趋势的偏差——是否忽视了某个正在变热的方向？

### 模块 11：冷启动 AI 工作流 — 从零构建知识库

这是平台从零启动的核心引擎。在没有任何用户和社区内容的情况下，系统自动构建一个完整的、有深度的公共知识库。

#### 11.1 冷启动三阶段

```
第一阶段：存量知识回溯（1-2周）
    ↓
第二阶段：实时热点接入（持续运行）
    ↓
第三阶段：知识库自检与补全（周期性）
```

#### 11.2 第一阶段：存量知识回溯

**目标**：一次性聚合市面上现有的 AI 知识，构建知识库的初始骨架。

**回溯范围**：

| 回溯维度 | 具体内容 | 数量目标 | 优先级 |
|---------|---------|---------|--------|
| **主流大模型** | GPT-4/4o/5, Claude 3.5/4/Opus, Gemini 2.5/3, DeepSeek V3/R1/R2, Qwen 2.5/3, Llama 3/4, Mistral, Kimi, 豆包, 文心, 通义等 | 前50个重要模型 | P0 |
| **重要框架工具** | LangChain, LlamaIndex, Haystack, Dify, Coze, AutoGPT, CrewAI, AutoGen, MCP协议, Ollama, vLLM, TGI等 | 前100个核心工具 | P0 |
| **关键论文** | 从Attention Is All You Need到最新，覆盖Transformer、RLHF、RAG、Agent、MoE、推理优化等领域 | 前500篇关键论文 | P0 |
| **基础技能** | Prompt Engineering, RAG, Fine-tuning, Agent设计, Workflow编排, 部署, 评测等 | 约30个核心技能 | P0 |
| **主流AI产品** | ChatGPT, Claude, Gemini, Copilot, Cursor, Perplexity, 豆包, Kimi, 文心一言等 | 前50个主要产品 | P1 |
| **重要事件** | 行业里程碑事件（GPT发布、开源模型突破、政策法规、重大收购等） | 前100个关键事件 | P1 |
| **关键人物/组织** | 核心研究者、公司、实验室、开源社区 | 前200个关键实体 | P1 |

**执行方式**：

```
1. 系统通过API/爬虫批量获取以上所有信息
2. 对每条信息进行：结构化提取 → 关键词分类 → 三AI质量审查 → 知识图谱入库
3. 建立初始的实体关联网络（引用关系、演化关系、竞争关系等）
4. 为每个核心实体生成初始技术报告
5. 建立初始技能树和场景库
```

**质量标准**：
- 回溯阶段的知识单元也必须通过三 AI 质量审查
- 对于有争议或信息不完整的部分，明确标注置信度和不确定性
- 时间信息必须准确，模糊的时间标注为"约XXXX年X月"或"时间待确认"

#### 11.3 第二阶段：实时热点接入

**目标**：冷启动完成后，立即切换到实时追踪模式，确保知识库与行业同步。

- 启动模块1的热点雷达，持续监控所有数据源
- 新的热点自动触发技术报告生成 + 三AI讨论 + 入库
- 新的信息自动关联、修正、更新已有的知识库实体
- **用户看到的是一个"一直在线"的知识库，感受不到它是一周前才启动的**

#### 11.4 第三阶段：知识库自检与补全

**目标**：周期性（每周/每月）由 AI 主动检查知识库的质量和完整度。

**自检项目**：

| 检查项 | 说明 |
|--------|------|
| **实体缺失** | 热门讨论中频繁出现但知识库尚未收录的模型/工具/论文 |
| **关联断裂** | 应该关联但没有关联的实体对（如论文与其实现代码未关联） |
| **信息过期** | 被新信息修正/推翻的旧内容，尚未更新 |
| **孤岛实体** | 与其他实体关联过少的知识单元，可能是分类或关联不到位 |
| **覆盖盲区** | 某个重要领域（如AI安全、AI芯片）的内容密度明显偏低 |
| **质量退化** | 之前评为A/B级的内容，现在可能因信息更新而降级，需复核 |

**补全动作**：
- 自动生成补全任务列表
- 按优先级自动执行信息采集和报告生成
- 对无法自动补全的（如信息不可获取），标注"待人工补充"

#### 11.5 冷启动的核心理念

- **知识库必须先于用户存在**。新用户注册时，面对的不是一个空壳，而是一个已经结构化的、有深度的 AI 知识体系。
- **用户可以是零，社区内容可以是零，知识库不能是零。**
- 冷启动阶段生成的内容全部标注来源和生成方式，透明化。

### 模块 12：开放生态

#### 12.1 开放 API
- 知识库实体可访问（"查询所有关于 RAG 的论文及其关联争议"）
- 评测数据可访问（"获取 GPT-5 在所有代码任务上的评分随时间的变化"）
- 讨论公开数据可访问（"某个话题下所有观点及其立场分布"）
- **用户私有数据永不对外开放**

#### 12.2 与其他平台的双向连接
- GitHub 集成：项目星标、issue、PR 动态实时同步
- ArXiv 集成：论文和讨论关联
- HuggingFace 集成：模型、数据集、Space 的更新和状态
- 社交媒体集成：关联讨论源头

#### 12.3 插件/MCP 生态
- 允许社区开发插件来扩展平台能力
- MCP Server 让 AI 可以通过标准协议接入外部能力

## 四、数据闭环——飞轮设计

```
热点捕获（代码公式过滤）
    ↓
三AI质量讨论 → 结构化 → 技术报告 → 公共知识库
    ↓                                       ↓
实时更新 ← 自动评测 ← 社区讨论（AI仅被召唤）  ← 知识库自检
    ↓                         ↓
个性化推荐 → 用户学习 → 用户个人知识库 → 用户贡献 → 公共知识库
    ↓

```

## 五、关键决策（已确定）

| # | 决策项 | 决策结论 |
|---|--------|---------|
| 1 | 内容质量 vs 速度 | **两者都要**。平台大部分决策由 AI 驱动，引入三个独立 AI 角色做对抗性质量讨论。代码公式 + 来源权重做新闻聚合质量控制。所有内容发布前必须经过三层质量审查。 |
| 2 | AI 参与讨论的边界 | **被召唤、中立角色**。AI 不主动发起讨论，不持有主观立场，仅在被用户 @召唤 时回应。严格遵守"AI 社区铁律"。 |
| 3 | 开放与隐私 | **知识库公开、讨论公开、用户数据严格隐私保护**。个人知识库加密存储，用户完全控制自己的数据。 |
| 4 | 起步策略 | **从"大模型应用开发"领域切入**。覆盖 LLM 应用开发的全技术栈（模型选型 → Prompt → RAG → Agent → 部署 → 评测），这是当前覆盖面最大、用户最多、内容更新最快的领域。 |
| 5 | 社区从零冷启动 | **AI 工作流全自动启动**。三阶段冷启动：存量知识回溯 → 实时热点接入 → 知识库自检补全。知识库必须先于用户和社区存在。 |
| 6 | 个人知识管理 | **新增核心模块**。用户拥有完全自主的个人知识库空间，可创建、收藏、管理、导出。与公共知识库双向链接，但隐私完全隔离。 |

## 六、起步策略详解：大模型应用开发

建议第一阶段聚焦"大模型应用开发"领域，理由：

1. **覆盖面最大**：涵盖模型选型、Prompt Engineering、RAG、Agent、Function Calling、部署、评测、成本优化等
2. **更新最频繁**：每周都有新模型、新框架、新范式出现
3. **用户群体最大**：从入门者到资深工程师都有学习需求
4. **冷启动数据最丰富**：市面上已有大量可聚合的存量内容
